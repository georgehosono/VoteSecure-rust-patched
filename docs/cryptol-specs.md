# Review of Cryptol Modules relevant to E2E-VIV Cryptography

## Background

From the inception of this project until mid-2025, we had hypothesized
that writing a Cryptol specification of the core cryptography used in
our E2E-VIV protocol would have utility.

Historically, in other projects wherein we used Cryptol as a canonical
formal specification of cryptographic concepts, such a specification,
especially if it is *formally validated* (that is, theorems about the
protocol and its algorithms are proven for the specification), is
useful for two key purposes:

 1. simulating the protocol as a *digital twin*, thereby possibly
    using it for co-simulation or bi-simulation testing, and
 2. executing the protocol in order to generate Known-Answer-Tests
    (KATs) in the form of symbolic traces, which can be used to test a
    concrete implementation.

Moreover, writing the *formal, mechanized* specification of a large
amount of cryptography, as we were aiming to do in this project, is a
significant task in its own right.  So we had to ask, did we have
enough resources at the moment, and were the positive outcomes
enumerated above worth the effort, at this point in time?

In order to measure the *formal specification gap* between the
underlying cryptographic primitives we needed, versus those that were
available, Frank Zeyda reviewed the contents of Galois's Cryptol-Specs
project, the canonical, de facto standard cryptographic library used
by everyone to specify cryptographic systems.  What follows is his
high-level, and partial analysis.

It is partial because we judged in early Summer 2025 that, for two
main reasons, we were putting our work with Cryptol on "pause."  Those
reasons for the moment (the end of August, 2025) are:

 1. *our statement of work does not promise formal verification of
    any specifications or implementations*--while we *want* to
    formally verify the protocols, algorithms, and their
    implementations in this library, there are insufficient
    *resources* to do so at this time,
 2. the effort/resources tied up in writing the formal specifications
    in Cryptol was better spent on other, more practical tasks.

We do intend to return to modeling the protocol's algorithms, and
perhaps even its principals (as Abstract State Machines), in future
work, so we are leaving the partial specification of the underlying
cryptography which Frank started in the repository for others to see
and learn from, and to provide a beachhead for future R&D.

## A Partial Cryptographic Gap Analysis

Galois's [cryptol-specs](https://github.com/GaloisInc/cryptol-specs/) GitHub repository contains two folders, [Common](https://github.com/GaloisInc/cryptol-specs/tree/master/Common) and [Primitive](https://github.com/GaloisInc/cryptol-specs/tree/master/Primitive) that include Cryptol modules. We describe the content of each of them in a separate subsection. Our analysis is based on the repository content on November 21, 2025, commit hash [3cafa98](https://github.com/GaloisInc/cryptol-specs/commit/3cafa987002d22734bcf2f60b37402bc53626f0a).

Per [`README.md`](https://github.com/GaloisInc/cryptol-specs/blob/master/Common/README.md) of cryptol-specs: "Common code shared between various cryptographic primitives that may not be fundamentally cryptographic on its own."
